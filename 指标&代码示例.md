## 两个脚本算了什么，准不准

### 04_run_generation_eval.py — 单模式评测

跑一遍生成，输出一个 `CacheStats`，包含：

| 指标 | 白话含义 | 算得准不准 |
|------|---------|:---:|
| `cache_runtime_bytes` | KV 缓存实际占了多少内存 | **准** — 直接量张量大小，不是估算 |
| `cache_compressed_bytes` | archive 量化部分多少字节 | **准，但容易误解**（见下） |
| `peak_gpu_bytes` | 峰值 GPU 显存 | **准** — PyTorch 原生接口量的 |
| `kv_compression_ratio` | 压缩倍率 = baseline原始大小 / 实际大小 | **准** |
| `bytes_per_token` | 每个 token 的 KV 均摊字节 | **准** |
| `recent_ratio` / `archive_ratio` | recent/archive 占总 token 数比例 | **准** |
| `memory_overhead_ratio` | 峰值显存 / 缓存占用，看额外开销多大 | **准** |

### 08_compare_modes.py — 多模式对比

在 04 的基础上多了：

| 指标 | 白话含义 | 算得准不准 |
|------|---------|:---:|
| **token consistency** | 和 baseline 的生成结果逐 token 对比，一致率多少 | **准** — 直接比较 token id，不经 tokenizer |
| **PPL (perplexity)** | 逐 token 解码算困惑度 | **准** — 走了真实量化 cache 路径 |
| **ΔPPL** | 和 baseline PPL 的差值 | **准** |
| **压缩-质量效率比** | 每损失 1% 一致性换到多少倍压缩 | **准** — 就是上面两个指标的除法 |
| **速度 (tok/s)** | 生成速度 | **准** |

### 关于"准不准"的两个细节

**1. `cache_compressed_bytes` 容易让人看错**

报告里会显示：
```
runtime=200MB  compressed=80MB
```

你可能觉得"还有 120MB 没压缩？"——不是的。`compressed` 只算了 archive 里纯量化编码的字节，没算 recent 的 fp16 原始字节和 archive 的 k_norms 元数据。那 120MB 就是 recent + meta，并不是"漏压缩了"。

**2. hawp_only 的压缩率混了两种节省**

hawp_only 模式下 `kv_compression_ratio` 可能显示 2.0x，但这个 2x 是低秩投影（把 4096 维压到 512 维）带来的，不是量化带来的。和 hawp_quant 模式的 2x 含义不同。数字本身没算错，只是"功劳归谁"分不清。

### 一句话结论

**两个脚本算的所有指标都和实际运行一致，没有算错或脱节的地方。** 唯一要注意的就是 `compressed_bytes` 的数字别拿来和 `runtime_bytes` 相减然后说"有这么多没压缩"，那个差值是 recent fp16 + 元数据，不是遗漏。




以下是我对 HAWP-LAQ 框架经过代码验证的改进建议和风险点：

## 一、正确性 / 内存问题（Critical）

### 1. `estimate_num_bytes` 计算结果与实际一致，但 3-bit 未实现真正的 3-bit packing

**位置**: `turboquant.py:221-231` (quantize) / `turboquant.py:299-333` (estimate_num_bytes)

**实际代码逻辑**:
- `quantize()` 对 4-bit 调用 `pack_uint4`（2 值/字节），对 2-bit 调用 `pack_uint2`（4 值/字节），对 8-bit 直接存 `uint8`。
- `estimate_num_bytes()` 通过 `qx.q.nelement() * qx.q.element_size()` 计算 **pack 后张量的实际字节数**，不是按"理想 bit-packing 公式"估算。
- 因此 2-bit / 4-bit / 8-bit 的报告值与实际内存占用一致，不存在 1.7~3x 偏差。

**但仍有以下事实**:
- 3-bit 仍使用 4-bit nibble 格式存储（`elif self.bits == 3: q_out = pack_uint4(q_flat)`），尚未实现真正的 3-bit packing。这导致 3-bit 模式的实际存储密度与 4-bit 相同。

### 2. `packbits.py` 中的有符号 `pack_int4` 是未被引用的冗余函数

**位置**: `packbits.py:89-107` (`pack_int4` / `unpack_int4`)

**实际代码逻辑**:
- `packbits.py` 确实包含使用有符号 `[-8, 7]` 的 `pack_int4` / `unpack_int4`。
- 但在整个 HAWP-LAQ 代码库中，`pack_int4` / `unpack_int4` **从未被导入或调用**。`turboquant.py` 的 import 列表只有 `pack_uint4, unpack_uint4, pack_uint2, unpack_uint2, pack_bool, unpack_bool`。
- TurboQuant 实际走的是 `pack_uint4` / `unpack_uint4`（`packbits.py:16-35`），该函数处理无符号 `[0, 15]`，与量化时的 `clamp(0, 15)` 完全匹配。
- **结论**: 运行时不会出现"15 被截断为 7"的问题。`pack_int4` 是一个未被引用的冗余函数，不影响系统正确性，但存在被误用的潜在风险。

### 3. `LayerKVCache` 量化后 archive 中只保留量化数据，不保留原始 fp32 副本

**位置**: `latent_cache.py:84-92` (`_quantize_flat`) / `latent_cache.py:104-118` (`_demote_excess`) / `latent_cache.py:120-135` (`demote_to_archive`)

**实际代码逻辑**:
- `_quantize_flat` 只构造并返回 `_QuantChunk(k_qx, v_qx, T, k_norms)`。输入的 `k`/`v` 张量作为局部参数，在函数返回后失去引用，会被垃圾回收。
- `_demote_excess` / `demote_to_archive` 在将 recent token 送入 `_quantize_flat` 后，会执行 `self._recent_k.clear()` / `self._recent_v.clear()`。
- `HAWPAttention._quantize_to_chunk`（`attention_hawp.py:289-299`）同样只构造 `_QuantChunk` 返回，`_quant_archive_chunks` 中只存量化结果。
- **结论**: archive 中只有量化后的 chunk，没有原始 fp32/fp16 数据的副本。之前文档中"archive raw bytes = 51200B"的结论无法从当前代码中得到支持。

## 二、性能问题（Significant）

### 4. `HAWPAttention` 内置缓存使用 `torch.cat` 逐 token 增长 — O(T²) 拷贝

**位置**: `attention_hawp.py:307-309`

**验证结果**:
- `HAWPAttention._quant_cache_append` 每新增一个 token 都执行 `torch.cat([self._quant_recent_k, k_new], dim=1)`，触发整块内存重分配 + 全量拷贝。
- 序列长度 T 时累计拷贝量为 `1 + 2 + ... + T = O(T²)`。对于长序列生成（如 4K+ tokens），这会成为严重瓶颈。

**补充**: `LayerKVCache` 路径（`latent_cache.py:75-76`）使用 `list.append` 存单 token 张量，避免了逐 token cat，但 retrieve 时仍需 `torch.cat(self._recent_k, dim=0)`（`latent_cache.py:152`）。

### 5. `_compute_archive_k_logits_approx` 使用 per-head per-chunk Python 循环

**位置**: `attention_hawp.py:932-964`

对每个 KV head 和每个 archive chunk 独立调用 `approx_inner_product`。假设 32 个 KV head + 50 个 chunk，每步 decode 需要 **1600 次** Python 级别的循环迭代，每次还要临时构造 `TurboQuantizedTensor` / `TurboQuantProdResult` 对象。这些操作无法被 GPU 批量并行，长上下文场景下是主要 decode 延迟来源。

### 6. 多个 archive chunks 从不合并

**位置**: `attention_hawp.py:373-390` (`_quant_cache_get_kv`) / `attention_hawp.py:340-371` (`_merge_quantized`)

每次 decode 都要对所有 chunk 逐个反量化再 concat。chunk 数量随序列增长线性增加（每 `recent_window` 个 token 产生 1 个 chunk），导致 decode 越来越慢。

`_merge_quantized` 静态方法被标记为 `TODO(optional fast-path): Currently unused`，说明 chunk 合并机制尚未启用。

### 7. Drop 操作导致 full dequantize + requantize

**位置**: `attention_hawp.py:404-416` (`drop_oldest_from_archive`) / `attention_hawp.py:420-466` (`drop_least_important_from_archive`)

当需要丢弃某个 chunk 中的部分 token 时，必须：
1. 反量化整个 chunk
2. 按索引切片保留剩余部分
3. 对剩余部分重新量化

这带来两个问题：
- **计算开销大**：反量化+重新量化代价远超简单的索引切片
- **量化误差累积**：每次 requantize 引入新的量化噪声，多次 drop 后误差会持续叠加

## 三、设计限制 / 功能缺口

### 8. `CacheManager` 不支持 `k_dim != v_dim`（NotImplementedError）

**位置**: `cache_manager.py:61-69`

**验证结果**: 传入 `k_dim=32, v_dim=16` 直接抛出 `NotImplementedError`。但训练系统（`ProjectorTrainer`）和 projector artifacts 完全支持非对称 `r_k != r_v`，rank search 的候选列表也不排除非对称配置。训练出来的非对称 projector 在 `CacheManager` 路径无法使用。

**补充**: 底层 `LayerKVCache` 本身已支持 `k_dim != v_dim`（`_recent_k` / `_recent_v`、`_quantize_flat` 中的 `dk` / `dv` 都是独立处理的），但 `CacheManager` 在入口处硬拦截了这种情况。

### 9. Rank search 只探索对称 `r_k == r_v`

**位置**: `rank_search.py:148-149`

```python
_evaluate_rank(q, k, v, rank, rank, ...)  # 始终 (rank, rank)
```

非对称 rank（如 `r_k=16, r_v=64`）在 projector 训练和运行时都支持，但 rank search 从未探索。对于 K 需要更高压缩但 V 需要更高精度的场景，非对称配置可能显著更好。

## 总结优先级

| 优先级 | 问题 | 影响 |
|--------|------|------|
| **P1** | #4 torch.cat O(T²) | 长序列 decode 性能急剧退化 |
| **P1** | #5 per-head per-chunk 循环 | 长上下文 decode 慢 |
| **P2** | #6 chunk 不合并 | decode 延迟随序列增长 |
| **P2** | #7 drop dequantize+requantize | 误差累积 + 计算浪费 |
| **P2** | #8 CacheManager 不支持非对称 | 功能缺口 |
| **P2** | #9 rank search 仅对称 | 未充分利用非对称配置潜力 |
| **P2** | #1 3-bit 未实现 true packing | 3-bit 实际密度同 4-bit |
| **P3** | #2 pack_int4 冗余 | 未被引用，存在未来误用风险 |
